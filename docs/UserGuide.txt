DataScythe v0.1.0 - Byte-Level User and Operator Guide
============================================================

Release Boundary
----------------

This guide describes the promoted production DataScythe v0.1.0 native build set.
The production surfaces are:

  apps/cli/datascythe          Command-line erasure tool
  apps/gui/DataScythe          Qt 6 desktop GUI
  DataScythe.py                Compatibility launcher for the CLI

The checked-in Windows release archive is:

  releases/native/windows/DataScythe-0.1.0-win64.zip

The archive contains DataScythe.exe (GUI), datascythe-cli.exe (CLI), Qt 6.9.0
MinGW runtime DLLs, platform plugins, and README.txt. The CLI binary inside the
release zip is named datascythe-cli.exe to avoid a case collision with
DataScythe.exe on case-insensitive Windows file systems.

Linux and macOS native release artifacts are not yet promoted under
releases/native/. Source builds for those platforms are supported. See
LinuxRelease.txt for Linux-specific notes.

Version 0.1.0 implements GNU shred-inspired pass scheduling, partition metadata
wipe, NTFS alternate data stream shredding, post-erase verification, erasure
certificates, pre-flight checks, typed GUI confirmation, and SSD hardware secure
erase (NVMe Sanitize and ATA SECURITY ERASE on Windows). Current unreleased
source builds also include a Qt GUI drive clone tab for byte-for-byte physical
drive copies and an Analytics tab for selected-drive inspection. See
EraseEngine.txt, PassScheduling.txt, FileFormat.txt, and Confirmation.txt for
byte-level internals.


Source of Truth
---------------

This guide describes the native C++ implementation under src/ with platform
backends in platform/ and application entry points in apps/.

Authoritative implementation files:

  src/core/erase_engine.cpp       Overwrite pipeline, verification, progress
  src/core/drive_clone_engine.cpp Byte-for-byte physical drive clone pipeline
  src/core/pass_scheduler.cpp     GNU shred-style pass schedule builder
  src/core/pattern_generator.cpp  Fixed and random overwrite buffers
  src/core/partition_table.cpp    MBR/GPT metadata region wipe
  src/core/certificate.cpp        Erasure certificate export
  src/core/verification.cpp       Post-erase sample reads
  apps/cli/main.cpp               CLI argument parsing and dispatch
  apps/gui/main_window.cpp        Qt GUI workflow and confirmation gates
  src/platform/drive_analyzer.h   Drive analytics interface and data model
  platform/windows/               Windows drive enumeration, raw I/O, secure erase

The reference GNU coreutils shred implementation is preserved at
docs/reference/shred.c. Pass scheduling and fixed-pattern encoding follow that
reference. DataScythe does not ship GNU coreutils as a runtime dependency.

kAppVersion in src/core/version.h is 0.1.0. The CLI --version flag and GUI
window title report this value.


Release Signature Verification
------------------------------

Version 0.1.0 Windows release artifacts include detached PGP signatures. The
release-signing public key is:

  docs/DataScythe_Release_Signing_2026_pubkey.asc

Release signing key:

  DataScythe Release Signing (2026) <release@datascythe.local>
  Fingerprint: 72E7 D550 77BA 8D7D 425B DC76 6133 2D0E 78FE F1CF

Before trusting a signature, inspect the public key and confirm that the
fingerprint matches the value above:

  gpg --show-keys docs/DataScythe_Release_Signing_2026_pubkey.asc

To verify the signed checksum manifest:

  gpg --import docs/DataScythe_Release_Signing_2026_pubkey.asc
  gpg --verify docs/SHA256SUMS.asc docs/SHA256SUMS

To verify the Windows release archive:

  gpg --verify releases/native/windows/DataScythe-0.1.0-win64.zip.asc releases/native/windows/DataScythe-0.1.0-win64.zip

PGP signatures confirm that the artifact matches the key holder's signature.
They do not make Windows treat the binaries as Authenticode-signed. See
CodeSigning.txt for the separate local Authenticode certificate used by
maintainers.


Supported Erase Scope
---------------------

DataScythe v0.1.0 performs destructive erasure on targets the operating system
exposes to user-mode I/O. It does not scan remote backups, cloud sync folders,
database transaction logs, swap files on other volumes, or firmware-reserved
storage outside OS visibility.

Supported operation modes:

  DriveClone         GUI byte-for-byte copy from source physical drive to target
  FullDeviceWipe     Overwrite every OS-addressable byte on a physical device
  QuickZeroFill      Single all-zero pass across a physical device
  ShredVolume        Shred-style overwrite of a mounted volume path
  ShredFiles         Overwrite one or more regular files
  ShredDirectory     Overwrite all files in a directory tree
  SsdSecureErase     Hardware-assisted erase (NVMe Sanitize / ATA SECURITY ERASE)

Supported target classes:

  Windows physical drives     \\.\PhysicalDriveN
  Windows volume mount points C:\, D:\, etc.
  Windows file and folder paths
  Linux block devices           /dev/sdX, /dev/nvme0n1 (when built for Linux)
  macOS raw devices             platform-specific paths (when built for macOS)

The tool blocks erase of the detected system drive and system volumes. Drive
enumeration without Administrator privileges may report size_bytes = 0 on
Windows because IOCTL queries fail; always run drive-level operations elevated.


Initial GUI State
-----------------

On launch, the main window displays an Erase tab, a Drive clone tab, and an
Analytics tab:

  Title:     DataScythe - Secure Drive Eraser
  Header:    Administrator privileges are required for raw device access.

Erase tab:

  Drive table: empty until Refresh drives is clicked
  Passes spin box: 3 (range 1..100)
  Random passes: checked
  Final zero pass: checked
  Remove files after shred: checked
  Recursive folders: checked
  Verify after erase: unchecked
  Wipe partition metadata: checked
  Shred NTFS ADS: checked
  Auto-save certificate: checked

Erase action buttons:

  Refresh drives
  Full-device wipe
  Quick zero-fill
  Secure erase volume
  SSD hardware secure erase
  Secure erase file(s)
  Secure erase folder
  Cancel (enabled only during an active job)
  Export certificate
  Export log

Drive clone tab:

  Source drive table: populated by Refresh drives
  Target drive table: populated by Refresh drives
  Verify byte-for-byte after clone: checked
  Export clone report
  Clone source to target
  Cancel (enabled only during an active job)

Analytics tab:

  Drive table: populated by Refresh drives
  Analyze selected drive
  Export analytics report
  Read-only report panel for identity, classification, capacity, volumes,
  geometry, alignment, cache, TRIM/discard, and health-related platform data

Progress bar and status label remain at 0% / Ready until an operation starts.


Drive Enumeration
-----------------

Procedure (GUI):

  1. Run DataScythe.exe as Administrator on Windows.
  2. Click Refresh drives.
  3. Review physical index, device path, model, serial, size, type, and volumes.
  4. Confirm the system-drive flag is not set on the intended target.

Procedure (CLI):

  datascythe --list-drives

CLI output format per drive:

  Drive N: <device_path>
    Model:  <model string>
    Serial: <serial string>
    Size:   <size_bytes> bytes
    Type:   HDD | SSD | Removable | Virtual | Unknown
    System: YES | no
    Volumes: <space-separated mount points>

The physical_index field in drive metadata is the N used in the GUI confirmation
phrase ERASE N. It is not necessarily the same integer Windows Disk Management
displays; always match on device_path and model serial before confirming.


Drive Analytics
---------------

The Analytics tab inspects the selected drive without starting an erase or
clone operation.

Procedure (GUI):

  1. Run DataScythe.exe as Administrator on Windows for the richest raw-device
     property access.
  2. Click Refresh drives.
  3. Open the Analytics tab.
  4. Select a drive row.
  5. Click Analyze selected drive.
  6. Review the report panel.

Windows source builds query generic storage properties through DeviceIoControl,
including:

  - Device path, physical index, model, vendor, serial, and firmware revision
  - Enumerated type and SSD/non-rotational classification
  - System/removable/read-only flags
  - Mounted volume paths, filesystems, sizes, and system-volume flags
  - Bus type, removable media flag, and command queueing
  - Disk geometry and media type
  - Adapter transfer limits and bus version
  - Logical/physical sector sizes and alignment offsets
  - Seek penalty, which helps distinguish rotational from non-rotational media
  - TRIM/UNMAP support
  - Write cache type, enabled state, changeability, flush support, and
    write-through support

Linux source builds read available /sys/block properties such as rotational,
removable, read-only, logical/physical block size, discard settings, scheduler,
write cache, model, vendor, serial, and firmware revision.

macOS source builds include baseline DriveInfo fields and diskutil info when
available.

Health summary is conservative. Generic OS storage properties can identify many
SSD/HDD and media behavior traits, but vendor SMART attributes, NVMe SMART /
health log counters, lifetime writes, temperature, spare percentage, and error
logs may require vendor-specific commands or external tools such as smartctl or
nvme-cli. When those counters are not exposed through the generic analyzer, the
report says so rather than inventing a health verdict.


Erase Configuration Defaults
----------------------------

EraseConfig defaults in src/core/erase_config.h:

  mode                         FullDeviceWipe
  pass_count                   3
  use_random_passes            true
  final_zero_pass              true
  round_file_size_to_block     true
  remove_after_shred           true
  recursive                    true
  verify_after_erase           false
  shred_alternate_data_streams true
  wipe_partition_metadata      true

The GUI maps these fields to the configuration group checkboxes and the passes
spin box. The CLI maps them to --passes, --no-random, --no-zero, --remove,
--no-remove, --recursive, --no-recursive, --verify, --no-partition-wipe, and
--no-ads.


Pre-Flight Checks
-----------------

Before any destructive operation, DataScythe runs platform pre-flight checks.
On Windows, drive-level modes require Administrator elevation.

Blocking errors include:

  - Administrator privileges missing for raw device or volume erase
  - System drive or system volume selected
  - Target media reported read-only
  - Target path does not exist (file/folder modes)

Warnings include:

  - Mounted volumes on a drive targeted for full-device wipe or SSD secure erase
  - Non-SSD drive selected for SSD hardware secure erase

The GUI shows blocking errors in a critical message box and stops. Warnings
present a Yes/No continuation dialog. The CLI prints the formatted pre-flight
report to stderr and exits with code 1 when pre-flight fails.


Confirmation Gates
------------------

DataScythe never starts a destructive write without explicit operator
confirmation unless --yes is passed on the CLI.

GUI drive operations require:

  1. Typed phrase ERASE <physical_index> in the confirmation dialog.
  2. A final Yes/No dialog: "Last chance: permanently erase <device_path>?"

GUI drive clone operations require:

  1. Source and target drive rows selected in the Drive clone tab.
  2. Target is not the detected system drive, is not read-only, differs from
     source, and is at least as large as source.
  3. Typed phrase CLONE <target physical_index>.
  4. A final Yes/No dialog confirming overwrite of the target with the source.

GUI file operations require:

  1. Typed phrase SHRED FILES.
  2. Final Yes/No dialog for the selected path(s).

GUI folder operations require:

  1. Typed phrase SHRED FOLDER.
  2. Final Yes/No dialog for the folder path.

CLI operations require:

  Interactive stdin line exactly equal to YES (case-sensitive) unless --yes.

The drive clone workflow is GUI-only in current source builds; the CLI has no
clone mode.

See Confirmation.txt for normative phrase definitions and acceptance rules.


Full-Device Wipe
----------------

Overwrites every byte the OS exposes on the selected physical drive.

Pipeline:

  1. Pre-flight and confirmation.
  2. Best-effort volume dismount.
  3. Optional partition metadata wipe (first and last 1 MiB) when enabled.
  4. Pass schedule execution (pass_count passes, optional random mixing,
     optional final zero pass).
  5. Optional verification sample reads.
  6. Optional certificate export.

GUI: select drive row, click Full-device wipe.

CLI:

  datascythe --mode full --yes --confirm-target \\.\PhysicalDrive2 \\.\PhysicalDrive2

Use --passes N, --no-random, --no-zero, --verify, --no-partition-wipe, and
--certificate <path> to adjust behavior.


Quick Zero-Fill
---------------

Performs a single 0x00 overwrite across the device. Ignores pass_count and
random pass settings except for partition metadata handling and verification.

GUI: select drive, click Quick zero-fill.

CLI:

  datascythe --mode quick --yes --confirm-target \\.\PhysicalDrive2 \\.\PhysicalDrive2


Secure Erase Volume
-------------------

Shred-style overwrite of the first mounted volume on the selected physical
drive. The engine opens the volume mount point (for example D:\) rather than
the raw \\.\PhysicalDriveN path.

If the drive has no mounted volumes, the GUI shows "No volumes" and aborts.

GUI: select drive, click Secure erase volume.

CLI:

  datascythe --mode volume --yes --confirm-target \\.\PhysicalDrive2 \\.\PhysicalDrive2

The CLI resolves \\.\PhysicalDriveN to the first volume mount point when mode
is volume.


Drive Clone
-----------

Copies every OS-addressable byte from a selected source physical drive to a
selected target physical drive. This is intended for forensic imaging workflows
and full operating-system disk migration when the target is at least as large
as the source.

Pipeline:

  1. Operator selects one source drive and one target drive.
  2. Target safety checks reject same-drive, system-drive, read-only, and
     undersized targets.
  3. Typed clone confirmation and final Yes/No confirmation.
  4. Source and target raw devices are opened.
  5. Target volumes are dismounted best-effort.
  6. Bytes 0 .. source_size-1 are read from source and written to target.
  7. Target is flushed.
  8. Optional full byte-for-byte verification reads source and target back and
     compares every copied byte.
  9. SHA-256 hashes are recorded for the copied source stream and, when
     verification is enabled, the verified target stream.

GUI: open the Drive clone tab, select a source row, select a target row, leave
Verify byte-for-byte after clone checked for forensic use, choose whether to
zero-fill extra target space, then click Clone source to target.

If the target is larger than the source, bytes beyond source_size are either
left unchanged or zero-filled depending on the GUI option. Use zero-fill when
policy requires a fully defined target state.

Cloning a live system drive as the source is allowed but may not be
forensically stable because the running operating system can write to the
source during the copy. For evidence acquisition, use a powered-off source or a
trusted write-blocked environment whenever possible.

After a clone completes, click Export clone report to save a text report with
source and target identity, verification status, SHA-256 hashes, warnings, and
recent log lines. When verification is disabled, target SHA-256 is not available
because the target is not read back.


SSD Hardware Secure Erase
-------------------------

Issues firmware-level secure erase when supported:

  NVMe: Sanitize with BlockErase, falling back to CryptoErase
  ATA/SATA: SECURITY ERASE PREPARE (0xF3) then SECURITY ERASE UNIT (0xF4)

Requires Administrator on Windows. The drive must not be frozen, password-
locked, or under RAID management. ATA erase may require a power cycle before
the operation completes.

GUI: select drive, click SSD hardware secure erase.

CLI:

  datascythe --mode ssd-secure --yes --confirm-target \\.\PhysicalDrive1 \\.\PhysicalDrive1

Verification after SSD secure erase reads sample sectors expecting all-zero
bytes. Crypto erase or remapped blocks may cause verification failure even when
the sanitize command succeeded. See Limitations.txt.


Secure Erase Files
------------------

Overwrites selected files with the configured pass schedule. Optionally deletes
files after shredding (--remove / GUI checkbox). On Windows, NTFS alternate data
streams are shredded when shred_alternate_data_streams is enabled.

GUI: Secure erase file(s) opens a multi-select file dialog.

CLI:

  datascythe --mode files --remove --verify --yes C:\temp\a.txt C:\temp\b.doc


Secure Erase Folder
-------------------

Collects files under a directory (recursive by default) and shreds each file
using the ShredFiles pipeline.

GUI: Secure erase folder opens a directory picker.

CLI:

  datascythe --mode folder --recursive --yes C:\temp\shred_me


Progress and Cancellation
-------------------------

During overwrite operations the CLI prints a carriage-return progress line to
stderr:

  Pass <current>/<total> (<label>) <pass_percent>% overall <overall>% ETA <seconds>s

The GUI updates a progress bar and status label from the same EraseProgress
structure. ETA is estimated from overall bytes written and elapsed wall time.

Cancel behavior:

  GUI: click Cancel during an active job.
  Effect: cancel_requested is set; the engine stops between chunk writes.

Cancellation is best-effort. A partially overwritten target may be unreadable
or inconsistent. Do not assume a cancelled job left data intact.


Verification
------------

When verify_after_erase is enabled, the engine reads up to three 512-byte
samples per target at offsets 0, size/2, and size-512 (when large enough).

Quick zero-fill and final zero pass verify all-zero bytes.

Fixed-pattern final passes verify against the PatternGenerator buffer for the
final pattern type.

Random final passes skip byte comparison and emit a warning that verification
was skipped. verification_passed is set true in that case.

SSD hardware secure erase uses verify_target_zeroed, which expects zero bytes
at sample offsets. This may not reflect crypto-erase semantics.

Drive clone verification is different from erase verification: when Verify
byte-for-byte after clone is checked, the clone engine performs a full read
compare of every copied byte from source and target.

See EraseEngine.txt and Limitations.txt.


Erasure Certificates
--------------------

On successful completion, the GUI auto-saves a certificate when Auto-save
certificate is checked. Default path:

  Windows: %USERPROFILE%\Documents\DataScythe\Certificates\
  Linux/macOS: $HOME/Documents/DataScythe/Certificates/

Filename pattern:

  datascythe-certificate-<sanitized-target>-<YYYYMMDD-HHMMSS>.txt

The CLI exports a certificate only when --certificate <path> is supplied.

Certificate field layout is specified in FileFormat.txt.

Manual export: GUI Export certificate button after a completed job.


Log Export
----------

The GUI log view mirrors Logger entries. Export log writes UTF-8 text with LF
line endings:

  [<YYYY-MM-DD HH:MM:SS>] [<LEVEL>] <message>

Levels: INFO, WARN, ERROR.

Log lines are not signed. Treat exported logs as operational records, not
tamper-evident proof.


Analytics Report Export
-----------------------

After analyzing a drive in the Analytics tab, click Export analytics report to
save the displayed report text. Analytics reports contain hardware identifiers
such as serial numbers and should be handled as sensitive asset inventory.


CLI Reference
-------------

Usage:

  datascythe --list-drives
  datascythe [options] <target> [targets...]

Options:

  --mode MODE       full | quick | volume | files | folder | ssd-secure
                    (default: full)
  --passes N        Overwrite pass count (default: 3)
  --no-random       Disable random shred passes
  --no-zero         Skip final zero pass
  --remove          Delete files after shredding
  --no-remove       Keep files after shredding
  --recursive       Recurse into subdirectories (folder mode)
  --no-recursive    Do not recurse (folder mode)
  --verify          Read back sample sectors after erase
  --verify-mode M   sparse | full | percent (default: sparse)
  --verify-percent N  Verify approximately N percent of 512-byte samples
  --no-partition-wipe  Skip MBR/GPT metadata wipe on block devices
  --no-ads          Skip NTFS alternate data streams (file/folder modes)
  --certificate P   Export erasure certificate to path P on success
  --dry-run         Show pre-flight results and the planned operation only
  --confirm-target P  Required with --yes for drive-level erase targets
  --yes             Skip interactive YES confirmation
  --version         Show version information
  --help            Show help

Exit codes:

  0   Success
  1   Usage error, pre-flight failure, confirmation declined, or erase failure

Operational Safety Rules
------------------------

  1. Verify docs/SHA256SUMS and PGP signatures before running a release build.
  2. Match physical drive index, model, and serial before confirming.
  3. Never erase or clone onto the system drive.
  4. Disconnect or accept dismount of mounted volumes on whole-device operations.
  5. Store certificates and logs only on controlled media.
  6. Assume SSD wear-leveling, HPA/DCO, and firmware regions may retain data.
  7. Assume a compromised host can observe paths, filenames, and timing.
  8. For forensic clone source drives, prefer powered-off acquisition or a
     trusted write-blocked environment over live-system cloning.
  9. For unattended drive-level CLI runs, pair --yes with --confirm-target using
     the exact target path.

See OpSec.txt and Limitations.txt for the full operational security model.
